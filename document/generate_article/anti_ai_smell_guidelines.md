# AI臭さを排除するためのガイドライン

## 目的
生成AIを使用して記事を作成・修正する際に、「AI臭い」と感じさせる表現を排除し、人間が書いた自然な文章にするためのガイドライン。

---

## AI臭さの主な特徴（避けるべきパターン）

### 1. 箇条書きと太字の過剰使用

#### 問題点
- 箇条書きが多すぎる（連続した箇条書きが3つ以上続く）
- 行頭が太字で装飾されすぎている
- マークダウンの `**` 記号が未処理のまま残っている

#### 良くない例
```markdown
- **データベースの選択**: 適切なデータベースを選ぶことが重要です
- **パフォーマンスの最適化**: クエリを最適化することで速度が向上します
- **セキュリティの確保**: データを保護することが必須です
```

#### 良い例
```markdown
データベースを選択する際は、ワークロードの特性を考慮します。パフォーマンスが重要な場合は、インデックス戦略とクエリ最適化に注力しましょう。セキュリティについても、暗号化とアクセス制御を適切に設定する必要があります。
```

---

### 2. 見出しでのコロン多用

#### 問題点
- 見出しに不必要なコロン（:）が多用されている
- 英語圏のライティング文化の影響が強い

#### 良くない例
```markdown
## データベース選択: 重要な考慮事項
## パフォーマンス最適化: ベストプラクティス
## セキュリティ対策: 実装ガイド
```

#### 良い例
```markdown
## データベース選択の考慮事項
## パフォーマンス最適化のベストプラクティス
## セキュリティ対策の実装
```

---

### 3. 抽象的な体言止めの多用

#### 問題点
- 抽象名詞で文を終わらせることが多い
- 英語表現からの直訳のような不自然さ

#### 良くない例
```markdown
- データベース設計の最適化
- スケーラビリティの確保
- 一貫性の維持
```

#### 良い例
```markdown
- データベース設計を最適化します
- スケーラビリティを確保する必要があります
- 一貫性を維持することが重要です
```

---

### 4. 定型的なフレーズの連続

#### 問題点
- 「重要です」「必要です」「役立ちます」などの定型フレーズが繰り返される
- 「～することができます」という回りくどい表現

#### 良くない例
```markdown
Spannerは外部整合性を実現することができます。これは分散環境で重要です。TrueTimeを使用することで、グローバルな一貫性を確保することが可能です。
```

#### 良い例
```markdown
Spannerは外部整合性を実現します。分散環境では、TrueTimeを使ってグローバルな一貫性を確保します。
```

---

### 5. 過度に丁寧な接続詞

#### 問題点
- 「それでは」「さて」「では」などの接続詞が多すぎる
- 不自然に段落の先頭に配置される

#### 良くない例
```markdown
では、次にRaft replicationについて説明します。さて、Raftはどのように動作するのでしょうか。それでは詳しく見ていきましょう。
```

#### 良い例
```markdown
次にRaft replicationについて説明します。Raftはリーダーを選出し、データを複製します。
```

---

### 6. 機械的な段落構成

#### 問題点
- 「まず」「次に」「最後に」という機械的な構成
- 全ての段落が同じパターン

#### 良くない例
```markdown
まず、Oracleのシャーディング機能について説明します。
次に、Spannerの設計思想を見ていきます。
最後に、両者を比較します。
```

#### 良い例
```markdown
Oracleはディレクトリベースのシャーディングを採用しています。一方、SpannerはTabletという独自の分割方式を使います。両者の設計思想には明確な違いがあります。
```

---

### 7. 不自然な並列表現

#### 問題点
- 同じ文法構造が3つ以上連続する
- リズムが機械的で単調

#### 良くない例
```markdown
Oracleは高可用性を実現し、Spannerは外部整合性を保証し、MySQLはコミュニティ主導で開発されています。
```

#### 良い例
```markdown
Oracleは高可用性を実現します。Spannerが外部整合性を保証する一方、MySQLはコミュニティ主導で開発が進んでいます。
```

---

### 8. 感嘆符や過度な強調

#### 問題点
- 「！」の多用
- 「非常に」「極めて」などの強調語の過剰使用

#### 良くない例
```markdown
Spannerは非常に優れたデータベースです！極めて高い性能を実現します！
```

#### 良い例
```markdown
Spannerは高性能なデータベースです。グローバル分散環境でも低レイテンシを実現します。
```

---

### 9. 句点・読点の前後の不適切な半角スペース

#### 問題点
- 句点（。）や読点（、）の前後に不要な半角スペースが混在する
- 英語の句読点ルールが日本語に混入している
- 文章が不自然に見える

#### 良くない例
```markdown
Raftレプリケーションでは、トランザクションのコミット記録がリーダーと過半数のフォロワーによって同期的にログに書き込まれることで、ゼロデータロスを保証します 。しかし、実際のDML（データ変更）の適用自体は、トランザクションのレイテンシを最小限に抑えるため、フォロワーに非同期で行われます 。
```

#### 良い例
```markdown
Raftレプリケーションでは、トランザクションのコミット記録がリーダーと過半数のフォロワーによって同期的にログに書き込まれることで、ゼロデータロスを保証します。しかし、実際のDML（データ変更）の適用自体は、トランザクションのレイテンシを最小限に抑えるため、フォロワーに非同期で行われます。
```

#### チェックポイント
- 句点（。）の直前に半角スペースがないか
- 読点（、）の直前に半角スペースがないか
- 句点・読点の直後に不要な半角スペースがないか（改行前を除く）

---

## 自然な文章にするためのチェックリスト

### 構造面
- [ ] 箇条書きは適切な量か（1段落に3項目以内）
- [ ] 見出しに不要なコロンがないか
- [ ] 段落の長さにバリエーションがあるか
- [ ] 機械的な「まず・次に・最後に」構成になっていないか

### 表現面
- [ ] 体言止めを減らし、動詞で終わる文を増やしたか
- [ ] 「～することができます」を「～します」に簡潔化したか
- [ ] 「重要です」「必要です」などの定型フレーズが連続していないか
- [ ] 接続詞が自然に使われているか

### 内容面
- [ ] 具体例や数値が適切に含まれているか
- [ ] 抽象的な説明だけでなく、実践的な情報があるか
- [ ] 読者の疑問を想定した説明になっているか
- [ ] 専門用語に適切な説明があるか

### リズム面
- [ ] 短文と長文が混在しているか
- [ ] 同じ文末表現が3回以上連続していないか
- [ ] 読んだときに自然なリズムがあるか

## take-yodaの文体の特徴（目指すべきスタイル）

### 過去記事から抽出した文体的特徴

#### 1. 丁寧だが簡潔な説明
- 「～ます・です」調で統一
- 回りくどくない、ストレートな表現
- 技術的な正確性を損なわずに初心者にも理解しやすい

#### 2. 実践的な例の提示
- 具体的なユースケースを示す
- コード例やコマンド例が豊富
- 「たとえば」を使って具体化

#### 3. 段階的な構成
- 基礎から応用へスムーズに展開
- 前提知識を明示
- 「まず」「次に」を多用せず、自然な流れ

#### 4. 技術用語の適切な扱い
- 初出時に簡潔な説明を付ける
- 専門用語と日本語訳を併記
- 読者の理解度を想定した配慮

#### 5. バランスの取れた構成
- 箇条書きと文章のバランスが良い
- 図表を効果的に活用
- 情報の密度が適切

## AI生成文のレビュー手順

### Step 1: 全体構造の確認
1. 箇条書きが多すぎないか確認
2. 見出しのコロンをチェック
3. 段落の長さにバリエーションがあるか確認

### Step 2: 表現の自然化
1. 体言止めを動詞表現に変更
2. 「～することができます」を簡潔化
3. 定型フレーズの連続を解消

### Step 3: リズムの調整
1. 音読して不自然な箇所を特定
2. 文末表現にバリエーションを持たせる
3. 短文と長文を適切に配置

### Step 4: 内容の深化
1. 抽象的な説明に具体例を追加
2. 技術的な裏付けを確認
3. 読者の視点で理解しやすさをチェック

## 記事修正時の注意点

### やってはいけないこと
- ❌ 全ての箇条書きを削除する（適度な使用は可）
- ❌ 過度にカジュアルな表現に変更する
- ❌ 技術的な正確性を犠牲にする
- ❌ 意図的に誤字や崩れた文法を入れる

### やるべきこと
- ✅ 箇条書きと文章のバランスを取る
- ✅ 自然な日本語表現を心がける
- ✅ 技術的な正確性を維持する
- ✅ 読者の理解を最優先にする

## まとめ

AI生成文を自然な文章にするためには、以下の3点が重要です：

1. **機械的なパターンを排除する**: 箇条書き・コロン・体言止めの過剰使用を避ける
2. **自然な日本語表現を心がける**: 簡潔で読みやすい文章を目指す
3. **take-yodaの文体を意識する**: 過去記事のスタイルを参考に、個性を出す

このガイドラインに従って記事をレビュー・修正することで、AI臭さを排除し、読者にとって価値のある自然な記事を作成できます。
